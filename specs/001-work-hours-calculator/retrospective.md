# SDD 회고: 근무시간 계산기 크롬 익스텐션

> **SDD(Spec-Driven Development)**: 구현 전에 스펙을 먼저 작성하고, 스펙을 기준으로 구현하는 개발 방식

## 프로젝트 요약

| 항목 | 내용 |
|------|------|
| 기간 | 2026-03-23 ~ 2026-03-24 (2일) |
| 규모 | ~350줄, 6개 파일 |
| 스택 | Chrome Extension (Manifest V3), Vanilla JS |
| 목적 | 네이버 웍스의 부정확한 일평균 잔여시간 대신, 실제 의무 근로시간 기준 일평균 필요 근무시간 계산 |

```
스펙 작성 → 구현 계획 → 구현 → 테스트 → 문제 발견 → 수정 → 코드 리뷰 → 배포
```

|------|------|------|
| 1. 스펙 작성 | `/write-spec`으로 입출력, 계산 로직, 엣지케이스, 태스크 정의 | ~20분 |
| 2. 구현 계획 | 파일 구조, 구현 순서 결정 | ~10분 |

| 8. 배포 | GitHub public 레포 + README | ~5분 |

## 스펙이 도움이 된 부분

### 1. 계산 로직이 구현과 1:1 매핑됐다

스펙의 계산 공식이 그대로 코드로 옮겨졌다. "뭘 어떻게 계산하지?" 같은 고민 없이 바로 구현할 수 있었다.

**스펙:**
```
1. 소정근로일 = 해당 월 평일 수 - 공휴일 수
2. 의무 근로시간 = 소정근로일 × 8h
3. 남은 총 근무시간 = 의무 근로시간 - 누적 근무시간
```

**구현 (calculator.js):**
```js
const workdays = weekdays.filter((d) => !holidaySet.has(d));
return workdays.length * 8;  // 의무 근로시간
```

### 2. 입출력 정의가 UI 구조를 결정했다

스펙에 입력 5개, 출력 6개가 표로 정리되어 있어서 popup.html 구조가 바로 나왔다. 디자인 고민 없이 "이 값들을 보여주면 된다"로 진행.

### 3. Out of Scope이 범위 팽창을 막았다

스펙에 명시적으로 제외한 항목들:
- 최대 근로시간(170h) / 연장근무 계산
- 출퇴근 기록 상세 조회
- 연차/반차 별도 관리
- 여러 달 통계

구현 중 "이것도 넣으면 좋겠는데?" 싶은 순간이 있었지만, Out of Scope에 적혀 있어서 범위를 지킬 수 있었다.

### 4. 태스크 분리가 작업 순서를 정해줬다

T001~T007이 실제 구현 순서와 거의 일치했고, 각 태스크가 독립적이라 순차 진행이 매끄러웠다.

## 스펙이 부족했던 부분

### 1. 대상 페이지가 틀렸다

| | 스펙 | 실제 |
|---|---|---|
| 대상 URL | `workplace.worksmobile.com/my-space/work-statistics` | `home.worksmobile.com` |

스펙 작성 시 **실제 사용할 페이지를 열어보지 않고** 가정했다. 구현 후 첫 테스트에서 바로 발견되어 수정했지만, 스펙 단계에서 잡았어야 했다.

**발생한 비용:** content script URL 패턴 변경, popup URL 체크 수정, 에러 메시지 변경

### 2. 엣지케이스가 나열만 되고 결정이 안 됐다

**스펙에 적힌 내용:**
```
- [ ] 남은 근무일이 0일 때 (월말 마지막 근무일 이후)
- [ ] 이미 의무 근로시간을 초과했을 때 (남은 시간이 음수)
```

체크박스만 있고 **처리 방침이 없었다**. 구현에서는 `Math.max(0)`으로 최소 처리만 했고, "완료!" 같은 UX 대응은 빠졌다.

### 3. DOM 크롤링 전략이 스펙에 없었다

이 프로젝트의 핵심은 네이버 웍스 페이지에서 데이터를 읽어오는 것인데, **어떤 DOM 요소에서 어떤 패턴으로 추출하는지** 스펙에 정의되지 않았다. 실제 DOM 구조를 확인하지 않고 즉흥으로 TreeWalker + 정규식 조합을 만들었다.

### 4. 기술적 제약이 스펙에 없었다

`activeTab` 권한으로는 `tab.url`에 접근할 수 없다는 사실을 몰라서, 팝업이 에러 상태로 표시되는 문제가 발생했다. **플랫폼 API의 권한 모델**을 스펙 단계에서 확인했어야 했다.

### 5. 휴가(연차)가 입력으로 고려되지 않았다

스펙의 입력 테이블에 "남은 근무일"의 소스를 달력 계산으로만 정의했다. 네이버 웍스 페이지에 이미 휴가가 반영된 "N일" 값이 있다는 걸 몰랐고, 배포 다음 날 연차를 올려보고 나서야 발견했다.

**Out of Scope에 "연차/반차 별도 관리"를 적었지만**, 페이지에서 이미 제공하는 값을 활용하는 것은 별도 관리가 아니라 데이터 소스 선택의 문제였다.

## 근본 원인 분석

5가지 문제의 근본 원인은 하나로 수렴한다:

> **스펙 작성 시 "실제 환경"을 확인하지 않았다**

- 대상 페이지를 열어보지 않았다 → 잘못된 URL
- DOM 구조를 확인하지 않았다 → 즉흥 크롤링
- 페이지에 어떤 데이터가 있는지 확인하지 않았다 → 휴가 미반영
- API 권한 모델을 확인하지 않았다 → activeTab 이슈

스펙이 **"무엇을 만들지"는 잘 정의했지만, "실제로 가능한지"는 검증하지 않았다.**

## 개선 조치

회고 결과를 바탕으로 스펙 템플릿과 스킬 정의를 수정했다.

### 스펙 템플릿 변경 (spec-template.md)

**1. 입력 테이블에 "소스"와 "확인" 컬럼 추가**

Before:
```
| 필드 | 타입 | 규칙 |
```

After:
```
| 필드 | 타입 | 소스 | 확인 | 규칙 |
```
> "확인" = 실제 데이터 소스에서 해당 값이 존재하고 접근 가능한지 검증했는가

**2. 엣지케이스에 처리 방침 필수화**

Before:
```
- [ ] 남은 근무일이 0일 때
```

After:
```
- [ ] 남은 근무일이 0일 때 → "이번 달 근무 완료!" 메시지 표시
```

**3. "기술적 리스크" 섹션 신설**

```
## 기술적 리스크
- [ ] 외부 페이지 DOM 구조 — 셀렉터가 변경될 수 있음
- [ ] API 권한 모델 — 필요한 데이터에 접근 가능한지
```

### 스킬 정의 변경 (SKILL.md)

핵심 질문에 2개 카테고리 추가:

- **데이터 소스 검증**: 입력 데이터가 실제로 어디서 오는가, 해당 소스에 접근 가능한지 실물 확인했는가
- **기술적 리스크**: API 권한 제약, 외부 시스템 변경 가능성, 플랫폼별 동작 차이

| 5 | **Out of Scope ≠ 무시해도 됨** | 제외한 항목이 정말 불필요한지, 더 쉬운 대안은 없는지 재확인했는가? |

## SDD는 효과가 있었는가?

**있었다.** 다만 조건부로.

스펙이 잘 작동한 영역은 **내부 로직** (계산 공식, 입출력, 범위 제어)이었다. 이 부분은 스펙 → 구현이 거의 직선으로 진행됐다.

스펙이 부족했던 영역은 **외부 연동** (DOM 크롤링, API 권한, 실제 페이지 확인)이었다. 이 부분은 "만들어보고 부딪혀서 알게 된" 것들이다.

**결론: SDD의 가치는 "적는 것" 자체가 아니라, "적으려고 할 때 빈칸을 발견하는 것"에 있다.** 이번에는 빈칸을 채우지 않고 넘어갔고, 그 빈칸이 그대로 버그가 됐다. 템플릿에 체크박스를 추가한 건, 다음에는 그 빈칸을 넘어가지 않기 위한 장치다.
